### 업계 공통 패턴 (3자 합의)

1. **Git Worktree 격리**: 거의 모든 도구가 에이전트 작업을 worktree로 격리
2. **Attention Routing**: 최고의 UX는 "채팅 뷰어"가 아니라 "주의력 라우터" (waiting, blocked, failed, needs review)
3. **Durable Event Model**: 실시간 스트리밍보다 영구적 이벤트/히스토리 모델이 더 중요
4. **Orchestration ≠ Terminal**: 터미널 멀티플렉서는 기반(substrate)일 뿐, 데이터베이스가 아님
5. **Adapter-based Abstraction**: "모든 에이전트가 동일하다" 가정은 위험 — capability-based 추상화 필요

### 3자 합의 사항 (High Confidence)

#### 1. Frontend: Preact + HTM (빌드 스텝 없이)

**3자 합의**: Vanilla JS는 5-view 규모에서 한계. 하지만 React/Svelte는 빌드 파이프라인 필수라 현재 zero-build 철학과 충돌.

- **Claude**: Preact+HTM (3KB, CDN 로딩, 기존 static serving 유지)
- **Codex**: React 또는 Svelte 추천했으나...
- **Gemini 크로스 리뷰**: "Claude의 Preact+HTM이 이 유스케이스에 더 적합. 로컬 관리 콘솔에 heavy build step은 불필요한 마찰"
- **Codex 크로스 리뷰**: "프레임워크 논쟁은 코어 상태 머신이 완성된 후에. Preact+HTM이면 충분"

**결론**: Preact + HTM 채택. CDN 로딩, Phase 3에서 도입.

#### 2. Data Model: Task → Run → Session 계층

**3자 합의**: Task와 Session은 다른 객체. Task가 여러 Run을 생성하고, Run이 Session에 매핑.

- **Codex**: "Task는 계획 단위, Session은 증거(evidence). 가장 중요한 도메인 구분"
- **Gemini 크로스 리뷰**: "Codex가 Domain Modeling에서 승리. Task/Run/Session 계층 채택 필수"
- **Claude**: agent_runs 테이블, 비용 모델 정규화 필요

**결론**: 확장된 스키마 채택:
```
projects → tasks → runs → run_events
                 → approvals
                 → artifacts
                 → usage_events
+ agent_profiles (capability 기반)
+ external_sessions (OpenCode/Claude 세션 매핑)
```

#### 3. Execution: Worker/Orchestrator 프로세스 분리

**3자 합의**: 요청 핸들러에서 장기 실행 에이전트를 관리하면 안 됨.

- **Codex**: "API/UI와 Worker/Orchestrator 프로세스를 즉시 분리"
- **Gemini**: "orphan 프로세스 문제. 전용 worker 프로세스/오케스트레이터 필요"
- **Claude**: ExecutionEngine 추상화 (TmuxEngine + SubprocessEngine 폴백)

**결론**: Express는 API/SSE만 담당, 별도 Worker가 에이전트 실행/모니터링.

#### 4. SSE for Real-time (multiplexed)

**3자 합의**: SSE가 올바른 선택. 데이터는 서버→클라이언트가 대부분.

- **Claude**: 단일 `/api/events` 엔드포인트에 이벤트 타입 멀티플렉스
- **Codex**: event ID, replay-from-cursor, heartbeat 추가
- **Claude DX**: polling을 reconnection fallback으로 유지

**결론**: SSE 채택 + event ID/replay + 폴링 fallback.

#### 5. Git Worktree 격리 (필수)

**3자 합의**: 에이전트 간 파일 충돌 방지를 위해 worktree 격리가 업계 표준.

- **Gemini**: "agent environment isolation이 빠진 핵심 기능"
- **Codex**: "CCManager, Claude Squad, par, Agor 모두 worktree 사용"
- **Codex 크로스 리뷰**: "disposable worktree 또는 명시적 git checkpoint 선호"

**결론**: 에이전트 실행 시 자동 worktree 생성, Review 단계에서 merge/apply.

### 주요 논쟁점 (Disagreements)

#### 1. View 구조: 4개 vs 5개 vs 3개

| 입장 | 주장 |
|---|---|
| **Claude UX**: 4개 | Projects를 Sessions에 병합 (VS Code Explorer처럼 트리 구조) |
| **Codex**: 3개 먼저 | Dashboard, Board, Run Inspector만 먼저. Projects/Usage는 후순위 |
| **Gemini 크로스 리뷰**: 5개 유지 | "Projects는 top-level 필터여야 함. 멀티 레포 유저에게 필수" |

**권장**: Codex 방식 채택 — 핵심 3개(Dashboard, Board, Run Inspector)를 먼저 구현, Projects/Usage는 Phase 4에서.

#### 2. CLI Injection 위험도

| 입장 | 주장 |
|---|---|
| **Claude Architect**: HIGH | CLI injection 취약점, user content가 직접 인자로 전달 |
| **Gemini**: FALSE POSITIVE | `spawn`의 args array는 shell interpretation을 피함. 진짜 위험은 에이전트가 실행하는 content |

**권장**: Gemini가 기술적으로 맞지만, stdin 파이프로 전환하는 것이 best practice. Security-in-depth.

#### 3. Agent Registry 영속성

| 입장 | 주장 |
|---|---|
| **Claude Architect**: In-memory | taskId→AgentHandle 인메모리 맵 |
| **Gemini 크로스 리뷰**: SQLite | "장기 실행 작업에 인메모리만 쓰면 데이터 손실. SQLite에 persist 필요" |

**권장**: Gemini 방식 — SQLite에 run state를 persist하고, 서버 재시작 시 tmux reattach/reconcile.

### 3자가 공통으로 놓친 것 (크로스 리뷰에서 발견)

**Claude 크로스 리뷰가 발견한 Critical 이슈 (다른 2자 모두 놓침):**

1. **인증/권한 없음 (P0)**: Express 앱에 auth가 전혀 없음. 포트 4177에 접근 가능한 누구나 모든 세션 읽기/쓰기/삭제 가능
2. **폴링 스케일링 병목 (P0)**: `loadSessions()`가 매 API 호출마다 모든 세션 JSON을 디스크에서 읽음. 100+ 세션에서 O(n) 디스크 읽기. 캐싱 없음
3. **Silent Error 무시 (P1)**: `sessionService.js:87`에서 `catch (error) { continue; }` — 파싱 실패 세션이 무시됨
4. **Atomic Write 없음 (P1)**: `writeJson()`이 직접 `fs.writeFile` — 동시 요청 시 JSON 파손 가능

**Gemini 크로스 리뷰 발견:**

5. **Log Streaming & Backpressure**: 대량 에이전트 출력에 대한 스트리밍/버퍼링 전략 미흡
6. **Artifact Lifecycle**: node_modules, build 폴더, 임시 worktree 등 스토리지 관리 자동화 필요
7. **Session Fork/Checkpoint**: 에이전트 hallucination 시 롤백 후 재시작 기능

**Codex 크로스 리뷰 발견:**

8. **Crash Recovery**: 서버 재시작 후 tmux reattach, 스토리지 재스캔, orphan/dead run 처리
9. **Provenance Data**: regex/title 휴리스틱 대신 parent/child run과 handoff를 명시적으로 모델링 필요
10. **Testing Beyond CRUD**: adapter contract tests, migration tests, event-reducer tests 필요

**Claude 크로스 리뷰의 전략적 권고:**

> "Gemini 방향(하드닝)을 먼저 따르고, Codex 방향(플랫폼화)은 그 다음에."
> - 즉각적 갭은 운영적(인증, 캐싱, silent error, liveness)
> - 그 다음 Gemini식 하드닝(worktree, snapshot, HITL, cost control)
> - 그 기반이 갖춰진 후에 Codex식 플랫폼 확장(task hierarchy, approvals)

## Part 3: Synthesized Recommendations

### 기술 스택 (합의)

| Component | Choice | Rationale |
|---|---|---|
| Backend | Express.js | 기존 유지, 충분히 적합 |
| Database | SQLite (better-sqlite3, WAL mode) | 로컬 퍼스트, ACID, 복잡 쿼리 지원 |
| Frontend | Preact + HTM (CDN, no build) | 3자 합의. 경량 컴포넌트 + zero-build |
| Real-time | SSE (multiplexed) + polling fallback | 서버→클라이언트 단방향 적합 |
| Execution | tmux (ExecutionEngine 추상화) | 프로세스 persistence, SubprocessEngine 폴백 |
| Process | API/UI 프로세스 + Worker 프로세스 분리 | 장기 실행 에이전트 격리 |

### 데이터 모델 (합의)

```sql
-- Core hierarchy: Project → Task → Run → Events
projects(id, name, directory, description, color, budget_usd)
tasks(id, project_id, title, description, status, priority, sort_order)
runs(id, task_id, agent_profile_id, worktree_path, branch,
```
