## 1. 프롬프트 최적화

### 1.1 시스템 프롬프트 압축

| 기법 | 설명 | 예상 절감 |
|------|------|-----------|
| **불필요한 지시 제거** | 모델이 이미 잘 수행하는 행동에 대한 중복 지시 삭제 | 10-20% |
| **구조화된 포맷 사용** | 장문 설명 → 불릿/테이블로 전환, 동일 의미를 적은 토큰으로 전달 | 15-25% |
| **프롬프트 압축 도구** | LLMLingua 등으로 의미 보존하며 토큰 수 축소 (최대 20x 압축 보고) | 30-50% |
| **변수 템플릿화** | 반복되는 프롬프트를 템플릿으로 만들고 변수만 교체 | 20-30% |

### 1.2 불필요한 컨텍스트 제거

- **Few-shot 예제 최소화**: 0-shot → 1-shot으로도 충분한 경우 예제 제거 (예제당 100-500 토큰 절약)
- **출력 포맷 제한**: JSON schema 지정으로 불필요한 서술형 출력 억제 → output 토큰 30-50% 절감
- **`max_tokens` 적극 활용**: 응답 길이 상한 설정으로 과도한 출력 방지
- **Tool 정의 최적화**: 사용하지 않는 tool 제거 (tool당 ~300-700 토큰 오버헤드)

### 1.3 구현 난이도 및 효과 요약

| 항목 | 예상 절감 | 구현 난이도 | 비고 |
|------|-----------|-------------|------|
| 시스템 프롬프트 리팩토링 | 15-30% input 절감 | **낮음** | 일회성 작업, 주기적 리뷰 |
| 출력 포맷 제한 (JSON) | 30-50% output 절감 | **낮음** | structured output 활용 |
| LLMLingua 등 자동 압축 | 30-50% input 절감 | **중간** | 품질 검증 파이프라인 필요 |
| Tool 정의 정리 | 5-10% input 절감 | **낮음** | 미사용 tool 제거만으로 효과 |

## 2. 캐싱 전략

### 2.1 Anthropic Prompt Caching (네이티브)

Anthropic API가 제공하는 서버사이드 캐싱으로, 동일한 프롬프트 prefix를 재사용할 때 **입력 토큰 비용 90% 할인**.

#### 가격 구조

| 작업 | 배수 | 설명 |
|------|------|------|
| 5분 캐시 쓰기 | 1.25x base input | 기본값, 히트 시 자동 갱신 |
| 1시간 캐시 쓰기 | 2.0x base input | 빈도 낮은 접근에 적합 |
| **캐시 히트 (읽기)** | **0.1x base input** | **90% 할인** |

#### 모델별 최소 캐시 토큰

| 모델 | 최소 토큰 |
|------|-----------|
| Opus 4.6 / 4.5 | 4,096 |
| Sonnet 4.6 | 2,048 |
| Sonnet 4.5 / 4 | 1,024 |
| Haiku 4.5 | 4,096 |
| Haiku 3.5 / 3 | 2,048 |

#### 구현 방법

```python
# 자동 캐싱 (권장) — top-level cache_control 추가
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    cache_control={"type": "ephemeral"},  # 5분 TTL
    system="대규모 시스템 프롬프트...",
    messages=[...]
)

# 1시간 캐시
cache_control={"type": "ephemeral", "ttl": "1h"}
```

#### 최적 사용 패턴
- **시스템 프롬프트 + RAG 문서**: 시스템 프롬프트와 참조 문서를 캐시하고 사용자 메시지만 변경
- **멀티턴 대화**: 이전 대화 히스토리를 캐시하여 매 턴마다 재처리 비용 제거
- **Tool 정의 캐싱**: 자주 변경되지 않는 tool 정의를 별도 캐시 breakpoint로 설정

#### 손익분기점
- 5분 캐시: **1회 히트**만으로 손익분기 (1.25x 쓰기 → 0.1x 읽기)
- 1시간 캐시: **2회 히트**로 손익분기 (2.0x 쓰기 → 0.1x × 2)

### 2.2 Semantic Cache (응용 계층)

유사한 의미의 쿼리에 대해 이전 응답을 재사용하는 애플리케이션 레벨 캐싱.

| 도구/방법 | 설명 | 장단점 |
|-----------|------|--------|
| **Redis + Vector Search** | 쿼리 임베딩으로 유사 질문 매칭 | 유연하지만 임베딩 비용 발생 |
| **GPTCache** | 오픈소스 semantic cache 라이브러리 | 즉시 적용 가능, 커스터마이징 제한 |
| **Helicone 내장 캐시** | 프록시 레벨에서 자동 캐싱 | 20-30% 비용 절감 보고 |
| **Exact Match Cache** | 해시 기반 정확 매칭 | 가장 단순, FAQ 등에 효과적 |

#### 예상 효과
- FAQ, 반복 질문이 많은 서비스: **40-60% 요청 캐시 히트**
- 일반 대화형 서비스: **10-20% 캐시 히트**
- Anthropic prompt caching과 조합 시 input 비용 최대 **90-95% 절감**

### 2.3 구현 난이도 및 효과 요약

| 항목 | 예상 절감 | 구현 난이도 | 비고 |
|------|-----------|-------------|------|
| Anthropic Prompt Caching | input 비용 90% | **낮음** | API 파라미터 추가만으로 적용 |
| Exact Match Cache | 히트 시 100% (API 호출 자체 생략) | **낮음** | Redis/메모리 캐시 |
| Semantic Cache | 히트 시 100% | **중간-높음** | 임베딩 파이프라인, 유사도 임계값 튜닝 |

### 3.2 라우팅 전략

#### Tier 분류 기준

| Tier | 모델 | 적합 작업 | 예시 |
|------|------|-----------|------|
| **Tier 1** (간단) | Haiku 4.5 | 분류, 추출, 요약, FAQ, 감정 분석 | 카테고리 분류, 키워드 추출, 간단한 Q&A |
| **Tier 2** (보통) | Sonnet 4.6 | 일반 대화, 코드 생성, 문서 작성, RAG 응답 | 고객 지원, 콘텐츠 생성, 데이터 분석 |
| **Tier 3** (복잡) | Opus 4.6 | 복잡한 추론, 다단계 분석, 고난이도 코딩 | 아키텍처 설계, 법률 분석, 연구 |

#### 라우터 구현 방식

1. **키워드/규칙 기반**: 요청 길이, 특정 키워드, API 엔드포인트별 분류 → 구현 쉬움, 정확도 낮음
2. **분류기 기반**: 경량 ML 분류기로 복잡도 판별 → 중간 난이도, 높은 정확도
3. **LLM-as-Judge**: Haiku로 먼저 복잡도 판별 후 라우팅 → 추가 비용 발생하나 유연
4. **Cascade 패턴**: Haiku 시도 → 실패/불확실 시 Sonnet → Opus 순 에스컬레이션

#### 오픈소스 도구
- **[claude-router](https://github.com/0xrdan/claude-router)**: Claude 전용 라우터
- **RouteLLM**: 범용 LLM 라우팅 프레임워크
- **MindStudio**: 시각적 라우팅 설정 도구

## 4. 컨텍스트 윈도우 관리

### 4.1 대화 요약 (Summarization)

오래된 대화를 요약하여 전체 히스토리 대신 압축된 요약만 컨텍스트에 포함.

```
[전체 히스토리: 50K 토큰] → [요약: 2K 토큰] + [최근 5턴: 5K 토큰] = 7K 토큰
절감: 86%
```

**구현 방법:**
- N턴마다 Haiku로 이전 대화 요약 생성 (요약 비용은 미미)
- 요약 + 최근 N턴만 컨텍스트에 포함
- 중요 정보(사용자 이름, 결정 사항 등)는 별도 메타데이터로 보존

### 4.2 슬라이딩 윈도우

최근 K개의 메시지만 유지하고 나머지 삭제.

| 윈도우 크기 | 평균 컨텍스트 | 적합 용도 |
|-------------|--------------|-----------|
| 최근 3턴 | ~3K 토큰 | 단순 Q&A, 검색 |
| 최근 10턴 | ~10K 토큰 | 일반 대화 |
| 최근 20턴 + 요약 | ~15K 토큰 | 복잡한 다단계 작업 |

### 4.3 RAG (Retrieval Augmented Generation)

전체 문서를 컨텍스트에 넣는 대신, 관련 청크만 검색하여 삽입.

| 접근법 | 컨텍스트 크기 | 비용 | 정확도 |
|--------|--------------|------|--------|
| 전체 문서 삽입 | 50-200K 토큰 | 매우 높음 | 높음 |
| **RAG (top-5 청크)** | **2-5K 토큰** | **매우 낮음** | **보통-높음** |
| RAG + Reranking | 3-8K 토큰 | 낮음 | 높음 |

**최적 청크 사이즈**: 256-512 토큰 (너무 작으면 문맥 손실, 너무 크면 토큰 낭비)

#### RAG 구축 비용 vs 절감 효과
- 임베딩 비용: ~$0.10/MTok (OpenAI ada-002 기준), 전체 LLM 비용의 1-3%
- 벡터 DB 운영: Qdrant/Pinecone 등 월 $20-200
- **컨텍스트 토큰 절감: 90-97%** (200K → 5K)

### 4.4 구현 난이도 및 효과 요약

| 항목 | 예상 절감 | 구현 난이도 | 비고 |
|------|-----------|-------------|------|
| 슬라이딩 윈도우 | 30-60% input | **낮음** | 즉시 적용 가능 |
| 대화 요약 | 60-85% input | **중간** | 요약 품질 관리 필요 |
| RAG 도입 | 90-97% input | **높음** | 임베딩/벡터DB 인프라 필요 |
| 스마트 트렁케이션 | 20-40% input | **낮음** | 우선순위 기반 컨텍스트 삭제 |

## 5. 배치 처리 및 비동기 API

### 5.1 Anthropic Message Batches API

비동기로 대량 요청을 처리하며, **input/output 모두 50% 할인**.

#### 핵심 사양

| 항목 | 상세 |
|------|------|
| 할인율 | input/output 모두 **50%** |
| 배치 크기 | 최대 **10,000건/배치**, 최대 100,000건 동시 처리 |
| 처리 시간 | 대부분 **1시간 이내**, 최대 24시간 보장 |
| Rate Limit | 표준 API와 **별도** (배치가 표준 한도에 영향 없음) |
| 결과 조회 | 폴링 또는 완료 시 조회 |

#### 배치 가격표 (주요 모델)

| 모델 | Batch Input ($/MTok) | Batch Output ($/MTok) | 표준 대비 |
|------|---------------------|-----------------------|-----------|
| Opus 4.6 | $2.50 | $12.50 | 50% 할인 |
| Sonnet 4.6 | $1.50 | $7.50 | 50% 할인 |
| Haiku 4.5 | $0.50 | $2.50 | 50% 할인 |

#### 적합한 사용 사례

- 대규모 데이터 분류/라벨링
- 콘텐츠 모더레이션 (비실시간)
- 일괄 보고서/요약 생성
- 모델 평가 (eval) 파이프라인
- 야간 배치 분석 작업

#### Prompt Caching과 중첩 할인

**Batch API 50% + Prompt Caching 90% = 최대 95% 절감**

| 조합 | Input 비용 (Sonnet 기준) | 절감률 |
|------|--------------------------|--------|
| 표준 | $3.00/MTok | — |
| Batch만 | $1.50/MTok | 50% |
| Cache 히트만 | $0.30/MTok | 90% |
| **Batch + Cache 히트** | **$0.15/MTok** | **95%** |

### 5.2 큐 기반 아키텍처 설계

```
[사용자 요청] → [분류기] → 실시간? → 표준 API
                         └→ 비실시간? → 메시지 큐 → Batch API 집계 → 결과 저장 → 알림
```

**권장 패턴:**
1. 요청 수신 시 실시간 필요 여부 판별
2. 비실시간 요청은 Redis/SQS 큐에 적재
3. 일정 건수(예: 1,000건) 또는 일정 시간(예: 30분) 도달 시 배치 제출
4. 결과 수신 후 사용자에게 비동기 알림

### 5.3 구현 난이도 및 효과 요약

| 항목 | 예상 절감 | 구현 난이도 | 비고 |
|------|-----------|-------------|------|
| Batch API 직접 사용 | 50% | **낮음** | API 호출 방식만 변경 |
| 큐 기반 자동 배치 | 50% | **중간** | 메시지 큐 인프라 필요 |
| Batch + Prompt Caching | 최대 95% | **중간** | 두 기능 조합 설정 |

| 플랫폼 | 유형 | 특징 | 비용 |
|--------|------|------|------|
| **Helicone** | SaaS / OSS | 1줄 코드 설정 (base URL 변경), 자동 비용 추적, 내장 캐시 (20-30% 절감) | 무료 티어 + 유료 |
| **Langfuse** | OSS (셀프호스팅 가능) | 상세 트레이싱, 복잡한 워크플로우 디버깅, 사용자별 비용 분석 | 무료 (셀프호스팅) |
| **LangSmith** | SaaS | LangChain 생태계 통합, 평가 + 모니터링 | 유료 |
| **자체 구축** | 커스텀 | API 응답 usage 파싱 → DB 저장 → Grafana 대시보드 | 인프라 비용만 |

### 6.4 예산 관리 및 알림 아키텍처

```
[API 프록시/미들웨어] → 요청/응답 로깅 → [시계열 DB (InfluxDB/TimescaleDB)]
                                              ↓
                                    [대시보드 (Grafana)]
                                              ↓
                                    [알림 (Slack/Email/PagerDuty)]
```

**계층별 예산 제어:**

| 계층 | 제어 방식 | 초과 시 동작 |
|------|-----------|-------------|
| 전체 서비스 | 월 총 예산 한도 | 경고 → 서비스 degradation |
| 테넌트/고객별 | 일/월 토큰 한도 | 429 응답 + 알림 |
| 사용자별 | 요청당/일 토큰 한도 | Rate limit + 하위 모델 자동 전환 |
| 기능별 | 기능별 모델/토큰 할당 | 하위 모델 fallback |

### 6.5 이상 탐지

- **급증 감지**: 시간당 사용량이 7일 이동평균 대비 3σ 초과 시 알림
- **비용 이상**: 단일 요청이 $1 이상일 때 즉시 알림 (프롬프트 인젝션/루프 의심)
- **반복 패턴**: 동일 사용자가 동일 쿼리 반복 시 캐싱 미적용 경고

### 6.6 구현 난이도 및 효과 요약

| 항목 | 예상 절감 | 구현 난이도 | 비고 |
|------|-----------|-------------|------|
| Helicone 도입 | 20-30% (내장 캐시) + 가시성 | **낮음** | URL 변경 1줄 |
| 자체 미들웨어 로깅 | 간접 (최적화 근거 제공) | **중간** | 프록시 + DB + 대시보드 |
| 예산 제어 시스템 | 예산 초과 방지 | **중간-높음** | 멀티테넌트 과금 로직 |
| 이상 탐지 | 비정상 비용 방지 | **중간** | 통계 기반 알림 규칙 |
